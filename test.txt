Now the issue is different: you created a mixed Angular state.

Your package.json now asks for some Angular 18 packages, but your installed node_modules still contains Angular 17 packages. ng update is tripping over that mismatch, which is why it reports invalid: @angular-devkit/build-angular@17.3.17 ... instead of continuing. Angular’s update flow expects a consistent dependency tree, and the recommended upgrade path is still ng update @angular/cli@^18 @angular/core@^18.  ￼

Also, @angular-eslint major versions are meant to match the Angular CLI major version, so mixing Angular 17 CLI with Angular ESLint 18 packages is not a stable intermediate state.  ￼

What happened

You manually changed some packages to 18, for example:
	•	@angular-devkit/build-angular: ^18.0.0
	•	@angular-eslint/*: ^18.0.0

But these are still 17 in the same file or in installed modules:
	•	@angular/cli: ^17.3.17
	•	@angular/compiler-cli: ^17.3.12
	•	installed @angular-devkit/build-angular: 17.3.17

So right now your project is neither fully 17 nor fully 18.

Best fix from here

Do not continue with --allow-dirty.

First bring the repo back to a clean, internally consistent Angular 17 baseline. Then run the migration properly.

Step 1: undo the manual edits

Since you already made a backup commit, the safest move is to restore package.json and package-lock.json to the last clean state:

git restore package.json package-lock.json

If you changed more files and want to fully return to the last commit:

git reset --hard HEAD

Only do that if you are okay losing uncommitted edits.

Step 2: remove installed dependencies and lockfile

After restoring the Angular 17 state:

rm -rf node_modules package-lock.json
npm install

On Git Bash in Windows, that command is fine.

Step 3: verify you are back to clean Angular 17

Run:

npx ng version
npm ls @angular/cli @angular/core @angular-devkit/build-angular @angular-eslint/schematics

At this point, all Angular core/build packages should consistently show 17.x.

Step 4: commit this clean baseline

git add .
git commit -m "restore clean angular 17 baseline before migration"

Step 5: update the peer blockers first

Before ng update, update the packages that blocked you earlier:

npm install -D @angular-builders/jest@^18.0.0 @angular-eslint/builder@^18.0.0 @angular-eslint/eslint-plugin@^18.0.0 @angular-eslint/eslint-plugin-template@^18.0.0 @angular-eslint/schematics@^18.0.0 @angular-eslint/template-parser@^18.0.0

Angular 18 supports TypeScript >=5.4.0 <5.6.0 and RxJS ^6.5.3 || ^7.4.0, so your current typescript ~5.4.2 and rxjs ~7.8.1 are already within the supported range.  ￼

Step 6: now run the Angular upgrade

ng update @angular/cli@^18 @angular/core@^18

No --allow-dirty.

Step 7: if it succeeds, then install and test

npm install
ng version
ng build
npm test
npm run lint

Important rule for this migration

Do not manually bump random Angular packages one by one first.

For Angular migrations, keep these in sync as a set:
	•	@angular/cli
	•	@angular/core
	•	@angular/compiler-cli
	•	@angular-devkit/build-angular
	•	@angular-eslint/*

That is the key reason this run failed.
