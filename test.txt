Because you must use the same HTML for both PDF and DOCX, the safest fix is:

Stop using both of these

* native <ol><li> markers
* CSS layout hacks like table, table-cell, flex

Use one portable format for both

Convert ordered lists into plain paragraphs with marker text inside the content.

That avoids:

* DOCX adding its own bullets/numbers
* PDF and DOCX rendering differently
* unsupported CSS layout issues

⸻

Best single-format approach

Generate HTML like this:

<p style="margin:0 0 8px 0; padding-left:32px; text-indent:-32px;">
  <span style="font-weight:700;">1.</span> First item
</p>
<p style="margin:0 0 8px 32px; padding-left:40px; text-indent:-40px;">
  <span style="font-weight:700;">1.1.</span> Nested item
</p>

This works much better across:

* browser preview
* PDF renderers
* DOCX HTML importers

It is an old-school hanging-indent format, and DOCX engines usually handle it much better than flex or table.

⸻

What you should do

Instead of decorating <li> visually but keeping <ol>/<li>, you should replace ordered lists with paragraph blocks.

That means:

* no native list bullets
* no double numbering
* no flex/table
* same HTML for PDF and DOCX

1. Replace your preview builder with a portable builder

private buildPortableHtml(content: string): string {
  const root = document.createElement('div');
  root.innerHTML = content;
  root.style.fontFamily = 'Georgia, serif';
  root.style.fontSize = '16px';
  root.style.lineHeight = '1.6';
  root.style.color = '#1f2a37';
  this.applyElementStyles(root);
  this.transformOrderedListsToPortableBlocks(root);
  return root.innerHTML;
}

Use this same method for:

* PDF HTML
* DOCX HTML
* preview HTML

2. Add this main transformer

private transformOrderedListsToPortableBlocks(root: HTMLElement): void {
  const topLevelLists = Array.from(root.querySelectorAll('ol')).filter(
    (list) => !list.parentElement?.closest('ol')
  ) as HTMLOListElement[];
  for (const list of topLevelLists) {
    const fragment = document.createDocumentFragment();
    this.appendPortableOrderedList(list, [], fragment);
    list.replaceWith(fragment);
  }
}

3. Add recursive ordered-list converter

private appendPortableOrderedList(
  list: HTMLOListElement,
  parentPath: Array<{ index: number; style: OrderedListStyle }>,
  container: DocumentFragment | HTMLElement
): void {
  const listStyle = this.getOrderedListStyle(list);
  const start = parseInt(list.getAttribute('start') || '1', 10);
  const items = Array.from(list.children).filter(
    (child): child is HTMLLIElement => child instanceof HTMLLIElement
  );
  items.forEach((item, index) => {
    const currentIndex = start + index;
    const currentPath = [...parentPath, { index: currentIndex, style: listStyle }];
    const childNodes = Array.from(item.childNodes);
    const nestedLists = childNodes.filter(
      (node): node is HTMLOListElement =>
        node instanceof HTMLOListElement
    );
    const contentNodes = this.normalizePortableContentNodes(
      childNodes.filter((node) => !(node instanceof HTMLOListElement))
    );
    const paragraph = document.createElement('p');
    const level = currentPath.length - 1;
    const indent = 32 + level * 28; // tweak if needed
    const hanging = 32;
    paragraph.style.margin = `0 0 8px ${level * 28}px`;
    paragraph.style.paddingLeft = `${hanging}px`;
    paragraph.style.textIndent = `-${hanging}px`;
    paragraph.style.lineHeight = '1.6';
    paragraph.style.color = '#1f2a37';
    const marker = document.createElement('span');
    marker.textContent = `${this.formatMarkerPath(currentPath)}. `;
    marker.style.fontWeight = '700';
    marker.style.color = '#1f2a37';
    paragraph.appendChild(marker);
    if (contentNodes.length === 0) {
      paragraph.appendChild(document.createTextNode('\u00A0'));
    } else {
      for (const node of contentNodes) {
        paragraph.appendChild(node);
      }
    }
    container.appendChild(paragraph);
    for (const nestedList of nestedLists) {
      this.appendPortableOrderedList(nestedList, currentPath, container);
    }
  });
}

4. Replace your content normalizer with this

private normalizePortableContentNodes(nodes: ChildNode[]): ChildNode[] {
  const normalized: ChildNode[] = [];
  for (const node of nodes) {
    if (node.nodeType === Node.TEXT_NODE) {
      const text = node.textContent?.replace(/\s+/g, ' ').trim() ?? '';
      if (text) {
        normalized.push(document.createTextNode(text));
      }
      continue;
    }
    if (!(node instanceof HTMLElement)) {
      continue;
    }
    const tag = node.tagName.toLowerCase();
    // unwrap common block wrappers from TinyMCE
    if (tag === 'p' || tag === 'div') {
      const text = node.textContent?.replace(/\s+/g, ' ').trim() ?? '';
      if (text) {
        normalized.push(document.createTextNode(text));
      }
      continue;
    }
    // keep inline tags, but remove troublesome margins
    node.style.margin = '0';
    node.style.display = 'inline';
    normalized.push(node);
  }
  return normalized;
}

How to use it

Wherever you currently do:

const html = this.buildPreviewHtml(content);

change it to:

const html = this.buildPortableHtml(content);

And send that same HTML to:

* PDF generator
* DOCX generator

⸻

Why this fixes both

Your current bad DOCX output happens because Word sees:

* real list structure
* plus your custom visual number

So it renders both.

With the new portable HTML:

* there is no <ol> or <li> left after transformation
* number is plain text inside paragraph
* indentation is simple paragraph CSS
* DOCX and PDF both read the same thing

⸻

Tradeoff

This is the honest tradeoff:

* You gain consistent output in both PDF and DOCX
* You lose native editable Word list semantics

But since your external DOCX API is not reliably supporting your current list HTML, this is the most stable single-HTML solution.
