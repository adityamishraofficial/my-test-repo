# PrimeFlex v2 → v3 Migration: Audit, Fix, and Complete (Strict Guidelines)

## Context

This Angular codebase uses PrimeNG (UI component library) and PrimeFlex (CSS utility library). PrimeFlex was upgraded from v2 to v3, and a previous AI-assisted migration introduced multiple categories of errors that must be fixed. Additionally, any remaining v2 classes must be migrated properly.

**Previous mistakes that must NOT be repeated:**

1. PrimeNG component tags (`<p-fieldset>`, `<p-button>`, etc.) were incorrectly changed to native HTML tags.
2. Valid PrimeFlex v3 utility classes were incorrectly converted to invalid Tailwind-style shorthand (e.g., `justify-content-around` → `justify-around`, `align-items-center` → `items-center`). These shortened classes **do not exist in PrimeFlex** and silently break layouts.
3. v2 column classes (`p-md-2`) were incorrectly replaced with invented classes (`flex-md-2`) instead of the correct `md:col-2`.
4. v2 offset classes (`p-md-offset-1`) were incorrectly replaced with spacing utilities (`ms-md-1`) instead of the correct `md:col-offset-1`.

Your job is to audit, revert wrong changes, and complete the migration correctly.

## CRITICAL RULES — READ BEFORE TOUCHING ANY FILE

### Rule 1: PrimeFlex v3 is NOT Tailwind CSS

PrimeFlex v3 keeps **full CSS property names** in its class names. It deliberately did NOT adopt Tailwind's shorthand. These are the **correct, final v3 class names** — they must NEVER be shortened further:

| Correct v3 (KEEP AS-IS) | Wrong (Tailwind-style, DO NOT USE) |
|---|---|
| `justify-content-start` | ❌ `justify-start` |
| `justify-content-end` | ❌ `justify-end` |
| `justify-content-center` | ❌ `justify-center` |
| `justify-content-between` | ❌ `justify-between` |
| `justify-content-around` | ❌ `justify-around` |
| `justify-content-evenly` | ❌ `justify-evenly` |
| `align-items-start` | ❌ `items-start` |
| `align-items-end` | ❌ `items-end` |
| `align-items-center` | ❌ `items-center` |
| `align-items-baseline` | ❌ `items-baseline` |
| `align-items-stretch` | ❌ `items-stretch` |
| `align-self-start` | ❌ `self-start` |
| `align-self-end` | ❌ `self-end` |
| `align-self-center` | ❌ `self-center` |
| `align-self-baseline` | ❌ `self-baseline` |
| `align-self-stretch` | ❌ `self-stretch` |
| `align-content-start` | ❌ `content-start` |
| `align-content-end` | ❌ `content-end` |
| `align-content-center` | ❌ `content-center` |
| `align-content-between` | ❌ `content-between` |
| `align-content-around` | ❌ `content-around` |

**If you see any class in the "Correct v3" column already in the code, LEAVE IT ALONE.** It is already correct. Shortening it produces an invalid class that compiles fine but silently breaks the layout in the browser.

If you find yourself about to write `justify-around`, `items-center`, `justify-between`, `self-start`, `content-center`, or similar Tailwind shorthand — **STOP**. That is a Tailwind class, not PrimeFlex. Use the full name.

### Rule 3: Distinguishing PrimeNG vs PrimeFlex

- Between `<` and `>` (an HTML tag) → **PrimeNG, leave alone**.
- Lowercase `p` + uppercase letter as attribute (e.g., `pButton`) → **PrimeNG directive, leave alone**.
- CSS selector targeting a PrimeNG component (e.g., `.p-button`, `.p-dialog-header`) → **PrimeNG style, leave alone**.
- A token inside `class="..."`, `[ngClass]="..."`, `[class.xxx]="..."`, or a SCSS/CSS selector matching a PrimeFlex utility pattern → **PrimeFlex utility, migrate if v2**.

### Rule 4: When in doubt, do not change

Skip and add to a "needs manual review" list. **Breaking a PrimeNG component or a valid v3 class is far worse than leaving a v2 utility unmigrated.**

### Rule 5: Do not invent class names

Every class name you write must come from the official PrimeFlex v3 documentation reference table below. If a class is not in the reference table, **do not use it**. Specifically:

- `flex-md-2`, `flex-sm-6`, etc. → do NOT exist. Use `md:col-2`, `sm:col-6`.
- `ms-md-1`, `me-md-1`, etc. as column offsets → do NOT exist. Use `md:col-offset-1`.
- `justify-around`, `items-center`, etc. → do NOT exist in PrimeFlex. Use the full names.

Search for invalid patterns introduced by the previous migration when it tried to convert v2 grid classes:

| Wrong (revert FROM) | Correct (revert TO) |
|---|---|
| `flex-sm-{1..12}` | `sm:col-{n}` |
| `flex-md-{1..12}` | `md:col-{n}` |
| `flex-lg-{1..12}` | `lg:col-{n}` |
| `flex-xl-{1..12}` | `xl:col-{n}` |
| `ms-sm-{1..12}` (used as offset) | `sm:col-offset-{n}` |
| `ms-md-{1..12}` (used as offset) | `md:col-offset-{n}` |
| `ms-lg-{1..12}` (used as offset) | `lg:col-offset-{n}` |
| `ms-xl-{1..12}` (used as offset) | `xl:col-offset-{n}` |
| `me-{bp}-{n}` (used as offset) | `{bp}:col-offset-{n}` |
| Bare `flex` used as a grid container (replacing `p-grid`) | `grid` |

### Search regex

```
\b(flex|ms|me)-(sm|md|lg|xl)-([1-9]|1[0-2])\b
```

### Caveat for `flex` and `flex-{n}`

- `flex` alone is a **valid v3 class** for `display: flex`. Only revert to `grid` when context shows it replaced a former `p-grid` container with column children.
- `flex-{1..12}` (without a breakpoint) is a **valid v3 class** for `flex-grow: {n}`. Only revert to `col-{n}` when used inside a grid container as a column.
- When unsure, leave it and flag for manual review.

### Caveat for `ms-{n}` and `me-{n}`

- `ms-1` through `ms-8` (without breakpoint) are **valid v3 spacing classes** for `margin-inline-start`. Leave these alone.
- Only revert when the breakpoint pattern matches `ms-{bp}-{n}` AND the original v2 file (per git history) used `p-{bp}-offset-{n}`.

- `p-grid` → `grid`
- `p-col` → `col`
- `p-col-{n}` → `col-{n}` (n = 1–12, fixed, nogutter)
- `p-{bp}-{n}` → `{bp}:col-{n}` (e.g., `p-md-6` → `md:col-6`)
- `p-col-offset-{n}` → `col-offset-{n}`
- `p-{bp}-offset-{n}` → `{bp}:col-offset-{n}` (e.g., `p-md-offset-1` → `md:col-offset-1`)
- `p-col-nogutter` → `col-nogutter`
- `p-grid-nogutter` → `grid-nogutter`

**Display:**
- `p-d-flex` → `flex`
- `p-d-inline-flex` → `inline-flex`
- `p-d-block` → `block`
- `p-d-inline` → `inline`
- `p-d-inline-block` → `inline-block`
- `p-d-none` → `hidden`
- Responsive: `p-d-{bp}-flex` → `{bp}:flex`

### Application rules

1. Show before/after diffs grouped by file. Process in batches no larger than 10 files.
2. Multi-class strings: `class="p-grid p-jc-center p-mt-3"` → `class="grid justify-content-center mt-3"`.
3. `[ngClass]` bindings:

5. Update `.scss` / `.css` selectors targeting PrimeFlex v2 utility classes (e.g., `.p-grid { ... }` → `.grid { ... }`). Only touch selectors targeting PrimeFlex utilities, never PrimeNG component classes.
6. Skip and report ambiguous cases:
   - `class="flex"` on a non-grid element (likely valid `display: flex`).
   - `class="flex-1"` on a flex child (likely valid `flex-grow: 1`).
   - Custom app classes that happen to match a pattern.
   - String-concatenated class bindings constructed at runtime.

## Self-check before EVERY class change

Before writing any class replacement, ask yourself:

1. Is this class inside `class="..."`, `[ngClass]`, `[class.xxx]`, or a SCSS/CSS selector? (If no → leave alone.)
2. Is the existing class actually a PrimeFlex v2 utility, OR is it already-correct v3, OR a custom app class? (If not v2 → leave alone.)
3. Is my replacement class in the official v3 reference table above? (If no → do not write it.)
4. Am I about to shorten `justify-content-*`, `align-items-*`, `align-self-*`, or `align-content-*` to a Tailwind-style name? (If yes → **STOP, that's the wrong direction**.)
