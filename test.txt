# PrimeFlex v2 → v3 Migration: Audit, Fix, and Verify

## Context

A previous migration pass attempted to convert PrimeFlex v2 utility classes to PrimeFlex v3 across this Angular codebase, but it introduced **two categories of errors** that must be fixed:

1. **Wrong v3 class names** — e.g., `flex` used instead of `grid`, `flex-md-2` instead of `md:col-2`, `ms-md-1` instead of `md:col-offset-1`.
2. **Modified PrimeNG component tags** — e.g., `<p-fieldset>` was changed to `<fieldset>`, breaking the PrimeNG component.

Your job is to audit the entire codebase, **revert incorrect changes to PrimeNG components**, **fix incorrect class replacements**, and **complete any remaining v2 → v3 migration** properly.

### Rule 2: Rule of thumb for distinguishing PrimeNG vs PrimeFlex

- Between `<` and `>` (an HTML tag) → **PrimeNG component, leave alone**.
- Lowercase `p` followed by uppercase letter as an attribute (e.g., `pButton`) → **PrimeNG directive, leave alone**.
- A CSS selector targeting a PrimeNG component (e.g., `.p-button`, `.p-dialog-header`) → **PrimeNG style, leave alone**.
- A token inside `class="..."`, `[ngClass]="..."`, `[class.xxx]="..."`, or a SCSS/CSS selector matching a PrimeFlex utility pattern → **PrimeFlex utility, migrate it**.

### Rule 3: When in doubt, do not change

Skip the change and add it to a "needs manual review" list. **Breaking a PrimeNG component is far worse than leaving a v2 utility class unmigrated.**

## Step 1: Verify PrimeFlex setup

1. Open `package.json`. Confirm `primeflex` version is `^3.0.0` or higher. Report the exact version found.
2. Open `src/styles.scss` (and any other global stylesheet entry point referenced in `angular.json`). Confirm PrimeFlex is imported, e.g.:
   ```scss
   @import "~primeflex/primeflex.css";
   ```
   - The `~` prefix resolves to `node_modules` via webpack — correct for Angular CLI projects.
   - If the project uses Angular 17+ with no webpack `~` support, change to `@import "primeflex/primeflex.css";`.
   - Confirm the import is **after** the PrimeNG theme import (cascade matters).
3. Check `angular.json` `styles` array — if PrimeFlex is imported there instead of (or in addition to) `styles.scss`, flag duplication.
4. Check current import order in `styles.scss`. If PrimeFlex is imported **after** `styles/components/all`, warn that PrimeFlex utilities will override custom component styles. Recommend whether to keep this order (typical utility-first) or move PrimeFlex earlier, but **do not change** it without my confirmation.
5. Report all findings before making any class changes.

## Step 2: Revert incorrect PrimeNG component changes

Search the entire codebase for HTML files where the previous migration replaced PrimeNG component tags with native HTML tags. Look especially for:

| Incorrectly changed to | Should be reverted to |
|---|---|
| `<fieldset>` (where it was previously a PrimeNG component) | `<p-fieldset>` |

Use git history (`git log -p`, `git blame`) on suspect files to confirm what the original tag was.

For each reversion candidate, show me the diff **before** and **after** for review. Do not auto-apply.

## Step 3: Fix incorrect v3 class names

Grep the codebase (`.html`, `.component.html`, `.scss`, `.css`, `.ts` files including inline templates) for **incorrect class patterns** the previous migration may have introduced.

### Wrong → Right mapping table

| Incorrect pattern | Correct v3 | When to fix |
|---|---|---|
| `flex` (used as grid container) | `grid` | Only when it replaced `p-grid` and contains column children. `flex` alone is a valid v3 class for `display: flex`, so check context. |
| `flex-{sm,md,lg,xl}-{1..12}` (e.g., `flex-md-2`) | `{bp}:col-{n}` (e.g., `md:col-2`) | Always. `flex-md-2` is not a valid v3 class. |
| `flex-{1..12}` used for column width | `col-{n}` | Only when used inside a grid container as a column. `flex-{n}` is valid v3 for `flex-grow`, so confirm context. |
| `ms-{sm,md,lg,xl}-{1..12}` (e.g., `ms-md-1`) used as a column offset | `{bp}:col-offset-{n}` (e.g., `md:col-offset-1`) | Always. `ms-md-1` doesn't exist; `ms-1` exists as margin-inline-start (spacing scale), which is wrong for offsets. |
| `me-{bp}-{n}` used as offset | `{bp}:col-offset-{n}` | Same reasoning. |
| Dash-suffixed responsive utilities like `jc-md-center`, `ai-md-start`, `d-md-flex` | `md:justify-content-center`, `md:align-items-start`, `md:flex` | v3 uses colon prefix. |

For each replacement, show diff before/after for review.

### Application rules

1. Show before/after diffs grouped by file.
2. Handle multi-class strings: `class="p-grid p-justify-center p-mt-3"` → `class="grid justify-content-center mt-3"`.
3. Handle `[ngClass]` bindings carefully:
   - String literals: `[ngClass]="'p-grid'"` → `[ngClass]="'grid'"`
   - Object syntax: `[ngClass]="{ 'p-grid': cond }"` → `[ngClass]="{ 'grid': cond }"`
   - Array syntax: `[ngClass]="['p-grid', extra]"` → `[ngClass]="['grid', extra]"`
4. Handle `[class.xxx]` bindings: `[class.p-grid]="cond"` → `[class.grid]="cond"`.
5. Update `.scss` / `.css` selectors targeting PrimeFlex v2 utility classes (e.g., `.p-grid { ... }` → `.grid { ... }`), but **only** when the selector targets a PrimeFlex utility, not a PrimeNG component class.
6. Skip and report ambiguous cases:
   - `class="flex"` on a non-grid element (likely a valid `display: flex`).
   - `class="flex-1"` on a flex child (likely a valid `flex-grow: 1`).
   - Custom app classes that happen to match a pattern.
   - String-concatenated class bindings constructed at runtime.

## Workflow constraints

- Work on the current feature branch. **Do not commit or push** any changes.
- For every file, show a diff summary **before** writing changes. Wait for confirmation if more than 20 files would be touched in a single batch — split into smaller batches.
- If anything is ambiguous, **ask before proceeding**. Visual regressions from a bad PrimeFlex migration are hard to spot in code review.
